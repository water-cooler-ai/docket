# Project memory

## Public documentation boundary

- Never include ticket IDs, ticket names, planning labels such as “revision 8,”
  PR numbers, branch names, or other internal delivery references in public
  documentation, examples, code comments, release notes, or changelogs.
- Describe shipped behavior directly. If historical context is essential,
  reference only a public released version such as `v0.1.0`.
- Keep issue-to-change traceability in the PR and issue tracker, not in the
  repository documentation.
